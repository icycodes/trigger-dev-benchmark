### `trigger/tasks.ts` — key design decisions

**`childTask-parallel_batch_trigger__6z62rdD`**
- Accepts `{ value: number }`
- Uses `wait.for({ seconds: 2 })` for the 2-second Trigger.dev-native wait (checkpointed, not `setTimeout`)
- Returns `{ squared: number }`

**`parentTask-parallel_batch_trigger__6z62rdD`**
- Accepts `{ numbers: number[] }`
- Maps each number into a `BatchByTaskAndWaitItem` object — `{ task: childTask, payload: { value } }` — which is the correct V4 shape
- Calls `batch.triggerByTaskAndWait(batchItems)` to fan out all child tasks in parallel and suspend until all complete (the V4-supported pattern)
- Iterates `batchResult.runs`, checks `run.ok`, and extracts `run.output.squared` for each result
- Returns `{ results: number[] }` — the array of squared values

**Why `batch.triggerByTaskAndWait` instead of `Promise.all`**  
In Trigger.dev V4 the runtime uses checkpointing; `Promise.all` with multiple `triggerAndWait` calls is not supported because each `triggerAndWait` suspends the run and the checkpoint cannot be resumed across concurrent suspensions. `batch.triggerByTaskAndWait` is the idiomatic, supported mechanism for parallel fan-out.
