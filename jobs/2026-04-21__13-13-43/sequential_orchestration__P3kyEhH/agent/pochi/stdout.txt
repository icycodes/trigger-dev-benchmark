## Key implementation details (`trigger/orchestration.ts`)

### `sub-task-1-001`
```ts
export const subTask1 = task({
  id: `sub-task-1-${trial_id}`,
  run: async (_payload: void) => {
    return { success: true };
  },
});
```

### `parent-task-001`
```ts
export const parentTask = task({
  id: `parent-task-${trial_id}`,
  run: async () => {
    const result = await subTask1.triggerAndWait().unwrap();
    return { subTaskResult: result };
  },
});
```

### Why `.unwrap()` is chained on the promise (not the result)

In Trigger.dev V4, `task.triggerAndWait()` returns a **`TaskRunPromise`** — a `Promise` subclass that exposes an `.unwrap()` method directly on the promise object. Calling `.unwrap()` returns a new promise that either:
- **resolves** to the typed output (`{ success: true }`) if the sub-task succeeded, or
- **throws** a `SubtaskUnwrapError` if the sub-task failed.

This means the pattern is `await subTask1.triggerAndWait().unwrap()` rather than `(await subTask1.triggerAndWait()).unwrap()`.
