## Key files

### `trigger/batch.ts`
```ts
import { task, batch } from "@trigger.dev/sdk/v3";

const trial_id = "t1";

// Sub-task: accepts a single `id` and returns it
export const processItemTask = task({
  id: `process-item-${trial_id}`,           // → "process-item-t1"
  run: async (payload: { id: string }) => {
    return { id: payload.id };
  },
});

// Parent task: fans out all IDs in parallel using batch.triggerByTaskAndWait
export const batchProcessorTask = task({
  id: `batch-processor-${trial_id}`,        // → "batch-processor-t1"
  run: async (payload: { ids: string[] }) => {
    const items = payload.ids.map((id) => ({
      task: processItemTask,
      payload: { id },
    }));

    const results = await batch.triggerByTaskAndWait(items);

    return results.runs.map((run) => {
      if (run.ok) return run.output;
      throw new Error(`Run ${run.id} failed: ${String(run.error)}`);
    });
  },
});
```

### `trigger.config.ts`
Registers the `./trigger` directory with `defineConfig`, pointing Trigger.dev at the task files.

## How it works

1. **`batchProcessorTask`** receives `{ ids: string[] }` and maps each ID into a `{ task, payload }` item.
2. **`batch.triggerByTaskAndWait(items)`** dispatches all sub-tasks in parallel and waits for every run to complete — no `Promise.all` is used.
3. Each result in `results.runs` is checked for `run.ok`; successes return `run.output`, failures throw.
4. The parent returns the collected array of `{ id }` outputs.
