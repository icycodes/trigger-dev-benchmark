### What's in `trigger/media.ts`

| Task | ID | Role |
|------|----|------|
| `downloadTask` | `download-task` | Step 1 – fetches media from `sourceUrl` |
| `processTask` | `process-task` | Step 2 – transcodes / transforms the download |
| `uploadTask` | `upload-task` | Step 3 – pushes processed media to one destination |
| `mediaPipeline` | `media-pipeline` | Orchestrator – drives steps 1 → 2 → 3 |

#### Orchestration logic (`mediaPipeline`)

```
payload: { trial_id, sourceUrl, extraDestinations? }
```

1. **Step 1 – download** `downloadTask.triggerAndWait({ sourceUrl, trial_id })`  
   Throws on `!ok`.

2. **Step 2 – process** `processTask.triggerAndWait({ localPath, mimeType, trial_id })`  
   Throws on `!ok`.

3. **Step 3 – upload**
   - **Single destination** → `uploadTask.triggerAndWait(...)` (simple path).
   - **Multiple destinations** (`extraDestinations` supplied) →  
     `batch.triggerByTaskAndWait([...])` fires all `upload-task` runs in parallel; any individual failure is collected and rethrown as one combined error.

4. **`try / catch`** wraps the entire orchestration:
   - On any thrown error → `logger.error(...)` + returns `{ status: "failure", error }`.
   - On success → returns `{ status: "success", uploads: UploadResult[] }`.
