## Key design decisions

### `trigger/queue.ts`

```ts
// 1. Named queue тАФ global concurrency limit of 1
export const criticalQueue = queue({
  name: "critical-queue-${trial_id}",
  concurrencyLimit: 1,           // only ONE run active at a time
});

// 2. Task bound to the queue
export const exclusiveTask = task({
  id: "exclusive-task-${trial_id}",
  queue: criticalQueue,          // routes through the named queue
  run: async (payload: ExclusiveTaskPayload) => { ... },
});

// 3. Trigger helper тАФ sets concurrencyKey at trigger time
await exclusiveTask.trigger(payload, {
  concurrencyKey: payload.resourceId,   // per-resource sub-queue
});
```

### Two-layer concurrency control

| Layer | Mechanism | Effect |
|---|---|---|
| Queue | `concurrencyLimit: 1` on `criticalQueue` | At most 1 run from this queue executes globally |
| Per-resource | `concurrencyKey: payload.resourceId` (trigger-time) | At most 1 run per unique `resourceId` executes at a time |

> **V4 note:** `concurrencyKey` is a **trigger-time** option (`TriggerOptions`), not a task-definition property. It is passed as the second argument to `.trigger()` / `.batchTrigger()`, which is why the `triggerExclusiveTask` helper is provided as the canonical entry point.
